# Case Study — 대규모 ML 배치 파이프라인 리팩터링: Airflow Worker 점유 99% 감소

> 글로벌 전자기업 마케팅 데이터 조직 | ML Engineer (리팩터링·최적화 주도) | 2026
> ※ 고객사 보안을 위해 구체 정보는 일반화했으며, 모든 다이어그램·수치는 재구성한 것입니다.

## 1. 배경

고객사는 마케팅 캠페인용 ML 모델 수십 개를 GCP 위에서 운영하고 있었다.
Cloud Composer(Airflow)가 배치를 오케스트레이션하고, 모델 학습·추론은 Vertex AI Custom Job으로 실행되는 구조였다.

## 2. 문제

운영을 인수하면서 세 가지 구조적 문제를 발견했다.

**(1) Worker 리소스 낭비** — Vertex AI Job의 완료를 확인하는 폴링 태스크가 Job이 끝날 때까지 Airflow Worker 슬롯을 점유했다. 모델에 따라 2~12시간. 동시 실행 배치가 늘어나면 Worker가 폴링만으로 가득 차, 정작 실행할 태스크가 큐에서 대기하는 상황이 발생했다.

**(2) 중복 코드** — 동일한 유틸리티 함수가 5개 이상의 DAG 파일에 복사되어 있었고, 설정값이 파일마다 하드코딩되어 있었다. 수정 하나가 N개 파일 수정을 의미했다.

**(3) 테스트 부재** — 테스트 환경이 없어 모든 변경이 운영에 직배포되고 있었다.

## 3. 접근

### 3-1. Worker 점유: 폴링 → reschedule 센서

Airflow Sensor의 두 가지 모드를 분석했다.

| 모드 | 동작 | Worker 슬롯 |
|---|---|---|
| poke (기존 폴링과 동일) | 조건 충족까지 슬롯을 잡고 대기 | Job 시간 내내 점유 |
| reschedule | 확인 후 슬롯 반납, 다음 확인 시각에 재스케줄 | 확인하는 순간만 점유 |

폴링 태스크를 `VertexAICustomJobSensor(mode='reschedule')`로 전환했다.
트레이드오프도 검토했다: reschedule은 재스케줄링 오버헤드로 완료 감지가 poke_interval만큼 늦어질 수 있다. 그러나 대상 Job이 시간 단위로 실행되는 배치라 분 단위 지연은 무의미했고, Worker 슬롯 확보 이득이 압도적이었다.

### 3-2. 구조: 공통 모듈 + DAG 팩토리

- 중복 함수를 `common/` 공통 모듈 5종(config, vertex, bigquery, gcs, airflow utils)으로 분리
- 주차별로 나뉘어 있던 유사 DAG들을 `DAGConfig` dataclass + factory 패턴으로 통합 — 신규 배치는 설정 객체 하나만 추가하면 되는 구조
- 학습 전체 완료 후 추론을 시작하는 순차 구조로 변경해 동시 실행으로 인한 OOM 제거

### 3-3. 테스트: GCP 없이 로컬에서

GCP 의존성이 코드 전반에 얽혀 있어 로컬 테스트가 불가능했다. `sys.modules` 패치로 GCP SDK를 모킹하는 방식을 설계해 unittest 57개를 작성했고, 이 방식이 팀 표준으로 채택됐다.

## 4. 결과

| 항목 | 이전 | 이후 |
|---|---|---|
| Worker 점유 (폴링) | Job당 2~12시간 | 확인 순간만 (약 99% 감소) |
| 배치 파일 | 7개, 중복 함수 산재 | 5개 + 공통 모듈 1곳 |
| 테스트 | 없음 (운영 직배포) | 57개, 로컬 실행 |

전체 모델 44개 태스크 기준 검증: 24분, 실패 0건.

## 5. 배운 점

- 인프라 비용 문제처럼 보이는 것의 상당수는 실행 모델 설계 문제다. Worker 증설 없이 센서 모드 하나로 해결됐다.
- 리팩터링의 전제는 테스트다. 57개 테스트가 있었기에 구조 변경을 운영 리스크 없이 진행할 수 있었다.
